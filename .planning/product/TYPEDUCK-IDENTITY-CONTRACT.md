# TypeDuck Identity Contract

**Capture date:** 2026-06-23  
**Requirement traceability:** IDEN-02  
**Decision traceability:** D-01, D-02, D-03, D-04, D-05, D-06, D-07, D-15, D-16, D-17

## Purpose

This contract is the single authority for TypeDuck Windows IME product identity. Per D-04, downstream agents must not invent new product names, paths, GUIDs, registry locations, or release artifact names outside this contract without updating this file first.

Phase 1 is contract-only work. Per D-15 and D-16, this artifact documents target identity and source evidence without renaming production binaries, registry keys, protocol packages, installer files, source directories, or runtime paths. Per D-17, this plan contributes only IDEN-02 toward Phase 1 requirements.

## Status Definitions

| Status | Meaning |
|--------|---------|
| locked | Product decision is fixed for later implementation unless PROJECT.md or phase context changes. |
| proposed | Target value is the recommended TypeDuck value but needs human review before production implementation. |
| deferred | Target direction is documented, but exact implementation belongs to a later phase or may depend on an engine/protocol decision. |

## Channel Readiness

D-06 requires TypeDuck identity names to be channel-ready where cheap, while explicitly excluding beta/stable side-by-side implementation from Phase 1.

- Recommended stable channel base names are unsuffixed: `TypeDuckLauncher.exe`, `TypeDuckTextService.dll`, `TypeDuckIME`, `TypeDuckLauncherMutex`, and `\\.\pipe\<username>\TypeDuckIME\Launcher`.
- If beta or nightly channels are later approved, suffix only channel-specific names consistently, for example `TypeDuckIME.Beta`, `TypeDuckLauncherBeta.exe`, `TypeDuckLauncherBetaMutex`, and `typeduck-windows-ime-beta-setup.exe`.
- AppId, CLSID, and profile GUID values in this contract are proposed and pending human review. Do not implement them in production until Phase 3 confirms approval or updates this contract.
- Phase 1 does not implement beta/stable side-by-side support. Later side-by-side support requires separate AppId, CLSID, profile GUID, pipe, mutex, install path, data path, and release artifact entries.

## Verification Guidance for Later Phases

- Re-run source audits for each row before implementing it because the Moqi scaffold may change.
- Verify no production code, installer, protocol, source directory, registry, runtime path, or release workflow file is changed by this Phase 1 task.
- For zh-HK registration, verify Windows shows Chinese (Traditional, Hong Kong) and the bilingual display text `TypeDuck 粵語輸入法 / TypeDuck Cantonese IME`.
- For resource strings, inspect both file metadata and visible UI surfaces.
- For IPC identity, verify both TSF client and launcher use the same TypeDuck pipe name and that server identity hardening is handled in the security phase.
