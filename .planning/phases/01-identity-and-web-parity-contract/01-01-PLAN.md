## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Planning artifact to implementation agent | Later agents will treat these markdown contracts as product authority, so inaccurate identity values can propagate into installer, TSF, IPC, and release work. |
| Local evidence to planning artifact | Current scaffold values are read from source files; stale or unsourced entries can produce misleading migration instructions. |
| User typed content to diagnostics | Phase 1 does not change logs, but the identity and banned-surface contracts must preserve the later requirement to avoid raw typed content in diagnostics by default. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01-01 | Spoofing | TypeDuck pipe/mutex identity entries | mitigate | Contract pipe and mutex names separately from Moqi values so later security work can add TypeDuck namespaced IPC identity checks. |
| T-01-01-02 | Tampering | Identity contract | mitigate | Require current scaffold value, target TypeDuck value, affected files, and verification notes per D-03; verification checks required categories and citations. |
| T-01-01-03 | Information Disclosure | Log/data directory entries and banned cloud features | mitigate | Contract TypeDuck-owned log/data paths and document WebDAV/cloud clipboard as banned surfaces to prevent later leakage of typed content or clipboard data. |
| T-01-01-04 | Elevation of Privilege | Generic backend config-tool launch | mitigate | Banned-surface contract must flag backend-declared `configTool` launch as prohibited and require a fixed first-party TypeDuck settings entry point later. |
| T-01-01-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs are permitted or needed in this plan; executor must not add install tasks. |
</threat_model>
