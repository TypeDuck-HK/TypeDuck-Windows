## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local Web alpha to fixture artifacts | Runtime screenshots and source excerpts become future product truth, so provenance and limitation records must prevent stale or fabricated parity data. |
| Screenshot capture to repository artifacts | Candidate or settings screenshots might reveal typed content or local user data if uncontrolled samples are used. |
| Web source worktree to fixture metadata | Dirty source state can affect observed behavior; fixture must preserve commit and dirty status. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02-01 | Tampering | Fixture JSON and markdown | mitigate | Record source path, commit/hash, dirty status, URL, capture date, and limitations; verification checks required provenance fields. |
| T-01-02-02 | Information Disclosure | Screenshots and candidate samples | mitigate | Use controlled TypeDuck Web alpha sample data only; if screenshots risk exposing local typed content, omit capture and document the limitation. |
| T-01-02-03 | Repudiation | Fixture capture process | mitigate | Include attempted commands/tools, browser/viewport metadata, and exact unavailable-evidence notes so later reviewers can reproduce or refresh evidence. |
| T-01-02-04 | Information Disclosure | Cloud/AI/WebDAV scaffold leakage in fixture notes | mitigate | Fixture should document Web alpha parity only and must not import scaffold cloud, AI, or WebDAV behavior as acceptable TypeDuck behavior. |
